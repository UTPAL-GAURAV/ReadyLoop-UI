## Elevator Pitch
ReadyLoop helps users prepare for job interviews using AI (Claude). The user pastes a JD into a local Claude session â€” Claude builds a structured, company-aware interview plan, conducts mock rounds with adaptive follow-up questioning, scores each attempt, and saves everything to a backend. A UI dashboard displays progress and self-assessment data.

---

## Core Problem
Job seekers lack a personalized, interactive way to practice interviews. Static question lists don't give feedback. Hiring coaches are expensive. ReadyLoop fills that gap with a zero-cost, terminal-native interview loop powered by Claude.

## Architecture

### Three Repos, Three Layers

| Repo | Purpose | Hosting |
|------|---------|---------|
| `ReadyLoop-UI` | React dashboard â€” display only, no user input | Vercel (free) |
| `ReadyLoop-Backend` | Node.js REST API + PostgreSQL | Render (free) + Neon (free) |
| `ReadyLoop` | Agent repo â€” `CLAUDE.md`, `README.md`, `.env` | User clones locally |

### How It Works End-to-End
1. User clones `ReadyLoop`, adds their auth token to `.env`
2. User opens a Claude session (terminal or VS Code extension) in that repo
3. Claude reads `CLAUDE.md` â€” this is the intelligence layer
4. User pastes a JD â†’ Claude builds an interview plan â†’ saved to backend
5. User picks a round â†’ Claude conducts the mock interview interactively
6. Claude saves all attempt data to backend via REST API
7. User views progress, feedback, and self-assessment on the UI dashboard

### Key Architectural Principles
- **No intelligence in UI or backend.** All AI logic lives in `CLAUDE.md` instructions.
- **UI is display-only.** Zero user input on the frontend.
- **Backend is a dumb data store.** REST API only â€” save and retrieve, no business logic.
- **Claude calls the backend silently.** No announcements, no permission asks.
- **Auth token env var:** `GOOGLE_AUTH_TOKEN` in `.env`
- **Copy token UX:** shown in UI header immediately after Google login
- **Retry context:** Claude always surfaces missed points from the last attempt at the start of a retry â€” not blocking, just shown as context

### Key Signals for Question Generation
Claude uses all of the following when generating questions and calibrating depth:

| Signal | Examples | Impact |
|--------|---------|--------|
| **Role type** | Backend, Frontend, Fullstack, Data, ML, PM | Determines topic domain |
| **Level** | Intern, Junior, Mid, Senior, Staff, Principal | Determines depth and complexity |
| **Company type** | FAANG, mid-size product, consultancy/service (e.g. TCS), enterprise (e.g. SAP), startup | Drives interview culture and rigor |
| **Geography** | US, Europe, India, etc. | US FAANG = heavy LeetCode; Europe = practical coding + design; India service cos = aptitude + basic CS |
| **Domain** | Fintech, e-commerce, SaaS, etc. | Shapes system design and domain-specific technical questions |
| **JD specifics** | Tech stack, YOE asked, responsibilities listed | Direct tailoring of question topics and stack depth |

### Difficulty Distribution per Round
Each round stores a `difficulty_profile` alongside questions and rubrics. This captures:
- Expected difficulty distribution (e.g. "60% medium, 30% easy, 10% hard")
- Style (e.g. "practical coding over algorithmic", "competency-based behavioral", "whiteboard system design")
- Depth calibration rationale (e.g. "Mid-level SAP Germany: moderate DSA, no hard LeetCode")

This is generated by Claude at plan creation and stored â€” so scoring and feedback are always calibrated against the right bar, not a generic one.

## Core Feature: JD â†’ Interview Plan

### Step 1 â€” JD Ingestion
- User pastes JD into the Claude session
- Claude reads it and identifies: company, role, level
- Claude asks for clarification only if company or role is genuinely ambiguous â€” minimal back-and-forth, one question max
- Once clear, Claude proceeds without further confirmation

### Step 2 â€” Plan Generation
Claude generates a structured interview plan:
- **Company tile** â€” company name + role name
- **Round tiles** â€” one per interview round (e.g. DSA, LLD, Technical, HR/Managerial)
  - Round count and types are inferred from Claude's knowledge of the company + role + level
  - Each round has a **pre-generated question set** saved to the backend at plan creation time
  - Each question also stores an **evaluation rubric** â€” what a strong answer looks like, key concepts/keywords that must be covered, common gaps
  - This ensures consistent, fair scoring across sessions without re-computation

### Step 3 â€” Plan Gist
- Claude presents a summary of the plan: rounds, what each covers, rough difficulty
- Shown **once** at plan creation
- User can ask Claude to recap at any time, or view it on the UI
- Claude then asks: "Ready for [first/chosen round]?"

## Core Feature: Mock Interview Session

### Round Selection
- User picks which round to attempt â€” no enforced order
- Claude loads that round's pre-generated questions from the backend before starting

### Questioning Style
- Claude works through the pre-generated question list
- **Adaptive follow-ups allowed:** Claude probes weak spots, asks for clarification, grills on the same topic â€” but never derails into a new topic or unrelated question
- The scored questions stay within the pre-determined set; follow-ups are supplementary

### Attempt Completion & Scoring
After a round attempt, Claude:
- Scores the attempt with a **confidence score** (0â€“100%)
- Round tile status:
  - `Cleared` â€” â‰Ą80% confidence
  - `Failed` â€” <80% confidence (score shown)
  - `Not Attempted` â€” default
- Saves the full attempt record to the backend (see data model below)

### Retry / Replicate
- User can freely retry a failed (or cleared) round â€” no cooldown, no restriction
- **Replicate button** on the round tile: generates a *fresh* question set for that round
  - Claude re-reads the JD
  - Claude fetches all prior question sets for that round from the backend
  - New questions are guaranteed non-overlapping with all previous sets
  - Stored as a new attempt variant, not overwriting the original
- Prior attempt feedback is surfaced by Claude at the start of a retry (not enforced, but shown as context)

## Data Model (Draft)

### `job_applications`
- `id`, `user_id`, `company`, `role`, `level`, `jd_text`, `plan_gist`, `created_at`

### `interview_rounds`
- `id`, `job_application_id`, `round_type` (free-text, inferred by Claude), `order_index`, `status` (not_attempted/cleared/failed), `confidence_score`, `estimated_duration_minutes`, `question_count`

### `question_sets`
- `id`, `round_id`, `attempt_number`, `questions` (JSON array), `difficulty_profile` (JSON)
  - Each question: `{ question_text, evaluation_rubric, key_concepts[] }`
  - `difficulty_profile`: `{ distribution: { easy, medium, hard }, style, depth_calibration_rationale, geography, company_type, estimated_duration_minutes, question_count }`

### `round_attempts`
- `id`, `round_id`, `question_set_id`, `started_at`, `completed_at`, `confidence_score`, `status`

### `question_attempts`
- `id`, `round_attempt_id`, `question_id`, `user_answer` (verbatim/summarized), `strong_points` (array), `missed_points` (array), `interviewer_expectation` (what a good answer covers), `follow_up_count`

### `users`
- `id`, `email`, `name`, `google_id`, `created_at`

## `CLAUDE.md` Behavior (Agent Repo)

The `CLAUDE.md` in `ReadyLoop` will instruct Claude to:
- Read auth token from `.env` (`GOOGLE_AUTH_TOKEN`)
- Handle token expiry (401) â†’ tell user to refresh from dashboard â†’ wait â†’ retry
- Handle Render cold starts (503/timeout) â†’ retry silently
- Never reference local file paths
- Run all API calls silently
- Validate JSON structure before every API write (hard rule â€” never send malformed data)

### Session Resume
- User explicitly names what to resume: "resume SAP Backend" or by unique ID
- If multiple applications exist for the same company, each gets a **short unique ID** (e.g. `SAP-001`, `SAP-002`)
  - ID is shown on the UI tile for easy visual reference
  - User references by ID or descriptive name in the Claude session
- On session start Claude checks for in-progress applications and lists them if any exist

### Data Validation (Hard Rule)
Before every API write, Claude runs an internal validation step:
- Verifies JSON structure is complete and well-formed
- Checks all required fields are present and correctly typed
- Never sends a request with malformed or incomplete data
- This is a non-negotiable rule in `CLAUDE.md` â€” UI has no fallback for bad data

### Time & Question Calibration
- At the start of each round Claude states the estimated duration (e.g. "This round is typically 45â€“60 minutes")
- User self-tracks time â€” Claude does not monitor it
- Question count is calibrated to fit the round duration at plan creation time
- Both `estimated_duration_minutes` and `question_count` are stored in the round record
- UI displays estimated duration on each round tile before user starts

### Full Session Flow
- On JD paste: ingest â†’ clarify minimally â†’ generate plan â†’ save â†’ present gist
- On round start: fetch question set â†’ state duration â†’ conduct interview â†’ save attempt data
- On replicate: fetch all prior question sets â†’ generate non-overlapping set â†’ save
- On retry: surface missed points from last attempt (non-blocking) â†’ start round
